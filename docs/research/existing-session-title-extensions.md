# Research: existing Pi session-title extensions

## Summary

Yes—Pi already has several extensions that automatically derive a persisted session name from an early user prompt. The closest match to this repository's full goal set is **`@d3ara1n/pi-session-namer`**: it uses a cheap `utility` side-model, launches fire-and-forget on the first prompt, truncates input, and marks the attempt consumed before calling the model. However, no inspected package cleanly combines every stated requirement: first **substantive** prompt filtering, negligible bounded tokens/cost, very low latency, non-blocking execution, a dedicated inexpensive/no-reasoning model, exactly one attempt, and race-safe preservation of manual names.

The strongest prior art is therefore real and crowded, but there remains a narrower differentiation opportunity around strict one-shot semantics, administrative-prompt filtering, explicit output-token/cost budgets, and manual-name race safety.

## Findings

1. **Closest competitor: `@d3ara1n/pi-session-namer` matches most performance goals, but depends on another package and has a manual-name race.** It starts on `before_agent_start`, sets `hasNamed = true` before launching a detached async IIFE (so failure is not retried), sends at most 2,000 prompt characters through the `utility` role supplied by `pi-model-roles`, and documents “~0.5–1s” first-title latency and near-zero later overhead. It uses a 10-second timeout and falls back to a truncated prompt if generation fails. It preserves names present at `session_start`, but the detached completion calls `pi.setSessionName(name)` without rechecking; a user rename made while generation is pending could be overwritten. Its call does not set an explicit output-token ceiling in the namer itself, so “dozens of tokens” is intent rather than a hard package-level budget. Version `0.3.2`; source and README are present, but the package adds `@d3ara1n/pi-model-roles` as a runtime dependency. [README](https://github.com/d3ara1n/pi-extensions/blob/main/packages/pi-session-namer/README.md) · [event implementation](https://github.com/d3ara1n/pi-extensions/blob/main/packages/pi-session-namer/src/index.ts) · [model call](https://github.com/d3ara1n/pi-extensions/blob/main/packages/pi-session-namer/src/namer.ts) · [package metadata](https://github.com/d3ara1n/pi-extensions/blob/main/packages/pi-session-namer/package.json)

2. **`@ryan_nookpi/pi-extension-auto-name` is the closest mature zero/low-reasoning, non-blocking pattern, but it is not strictly one-attempt.** It runs a fire-and-forget IIFE from `before_agent_start`, defaults to the active session model but permits a dedicated configured model, explicitly defaults reasoning to `minimal`, limits output to 60 tokens, and aborts after 10 seconds. It checks for an existing name both before launching and before setting it, which is race-safer for manual naming. But if generation fails and the session remains unnamed, a later `before_agent_start` invokes it again; there is no per-session consumed-attempt flag. It also treats any nonempty prompt as eligible rather than detecting a substantive task. Version `0.3.4`, with an `index.test.ts` in the monorepo. [README](https://github.com/Jonghakseo/pi-extension/blob/main/packages/auto-name/README.md) · [event implementation](https://github.com/Jonghakseo/pi-extension/blob/main/packages/auto-name/index.ts) · [completion limits](https://github.com/Jonghakseo/pi-extension/blob/main/packages/auto-name/utils/short-label.ts) · [package metadata](https://github.com/Jonghakseo/pi-extension/blob/main/packages/auto-name/package.json)

3. **`@carter-mcalister/pi-auto-name` is an English-only fork of the preceding design, with the same retry gap.** It watches `before_agent_start`, uses the current model with `minimal` reasoning, a 60-token cap, and a 10-second timeout in a detached async call. It rechecks before setting and leaves failures unnamed, but that means later prompts can trigger another attempt. It skips subagent sessions and clips final titles to 30 characters. Version `0.2.0`; its package scripts include lint, typecheck, and tests. [README](https://github.com/CarterMcAlister/pi-packages/blob/main/packages/pi-auto-name/README.md) · [event implementation](https://github.com/CarterMcAlister/pi-packages/blob/main/packages/pi-auto-name/src/index.ts) · [completion limits](https://github.com/CarterMcAlister/pi-packages/blob/main/packages/pi-auto-name/src/short-label.ts) · [package metadata](https://github.com/CarterMcAlister/pi-packages/blob/main/packages/pi-auto-name/package.json)

4. **`pi-session-name` is first-prompt, bounded, and non-blocking, but uses the active model and deliberately retries three times.** On the first `input`, it freezes that exact text, starts a detached task, uses `completeSimple` with the active model and `maxTokens: 24`, and attempts up to three calls. It never switches to later prompts and checks an existing name before starting. It does not explicitly configure reasoning, does not filter commands/administrative input, and does not recheck the name immediately before setting, leaving a small pending-generation rename race. It also manages terminal-tab status, which is separate from persisted naming. Version `0.1.2`; Pi’s gallery reports a 6.1 KB, zero-dependency package and the repository includes tests. [Pi gallery](https://pi.dev/packages/pi-session-name?page=37) · [source](https://github.com/ttttmr/pi-session-name/blob/main/src/index.ts) · [package metadata](https://github.com/ttttmr/pi-session-name/blob/main/package.json)

5. **`pi-session-title` has the strongest “first real prompt” filtering and a tight token cap, but blocks prompt handling and can retry after failure.** It ignores extension-originated input and text beginning `/`, `!`, or `$`, captures at most 2,000 characters, uses the current model, and requests at most 30 output tokens. It calls `setSessionName(title, "auto")`, intended to yield to later user renames, and checks again before setting. However, both its `input` and fallback `before_agent_start` handlers `await` generation, adding model latency before the main agent proceeds. A transient error leaves `titleSettled` false, so a later prompt can retry using the frozen first message. Version `1.1.0`; the repository has CI, Vitest tests, typecheck, and build scripts, making it one of the more mature focused packages. [README](https://github.com/djdembeck/pi-session-title) · [implementation](https://github.com/djdembeck/pi-session-title/blob/main/extensions/index.ts) · [package metadata](https://github.com/djdembeck/pi-session-title/blob/main/package.json)

6. **`pi-auto-session-titles` is non-blocking, but can make two calls, does not bound output tokens, and has a manual-name race.** It launches from `before_agent_start` without returning its promise, defaults title reasoning to `minimal`, and supports a dedicated provider/model. It checks for an existing title only before launching asynchronous generation; because it does not recheck before setting the generated title, it can overwrite a manual rename made while generation is pending. Its validator may issue a second completion if the first title is too long, malformed, vague, or ungrounded; otherwise it falls back to prompt text. It also has no `maxTokens` option in the `complete` call. Resumed sessions with conversation messages are skipped. Version `1.0.2`; the small package has no test script in its manifest. [README](https://github.com/edxeth/pi-auto-session-titles) · [implementation](https://github.com/edxeth/pi-auto-session-titles/blob/main/index.ts) · [package metadata](https://github.com/edxeth/pi-auto-session-titles/blob/main/package.json)

7. **`pi-auto-rename` is a credible cheap-model competitor, but triggers after the first assistant response rather than immediately from the substantive prompt.** It defaults to `anthropic/claude-haiku-4-5`, allows model selection, uses only the first user message for automatic naming, sets `maxTokens: 128`, and marks `namingAttempted` before one call. `message_end` and `agent_end` handlers await `autoName`, but this occurs after response events rather than delaying initial agent startup. It checks `pi.getSessionName()` before and after the call, preserving manual names. Version `0.1.4`; it has release/typecheck/lint automation but no test script. [README](https://github.com/victor-software-house/pi-auto-rename) · [implementation](https://github.com/victor-software-house/pi-auto-rename/blob/main/index.ts) · [package metadata](https://github.com/victor-software-house/pi-auto-rename/blob/main/package.json)

8. **`@therynamo/pi-auto-session-name` is generic first-turn naming, not cost-conscious side-model naming.** It waits for `turn_end`, collects user messages, then awaits one call to the current session model with no explicit output-token or reasoning limit. It does not check for an existing/manual name before setting, so a pre-set name can be overwritten. Its `hasNamed` flag becomes true only after successful generation, allowing retries on later turns after failure. Version `0.1.0`, with no tests or scripts in the package manifest. [README](https://github.com/therynamo/pi-auto-session-name) · [implementation](https://github.com/therynamo/pi-auto-session-name/blob/main/extensions/index.ts) · [package metadata](https://github.com/therynamo/pi-auto-session-name/blob/main/package.json)

9. **Feature-rich naming packages solve a broader problem and conflict with the strict cost/one-attempt goal.** `@furbyhaxx/pi-session-naming` defaults to three retries, may fall back once to the current session model, sends the whole active branch (including tool results by default), and deliberately postpones vague-prompt naming until later meaningful activity. It does preserve manual titles and can auto-select a lightweight model, but its conventional-tag formatting, retries, project metadata, browsing, and rename features make it a broader product rather than a minimal one-shot competitor. Its repository has extensive focused test files and a changelog. [Pi gallery](https://pi.dev/packages/@furbyhaxx/pi-session-naming) · [repository](https://github.com/furbyhaxx/pi-session-naming)

10. **`pi-sessions` is session-management/search software with recurring auto-retitling, not a negligible-cost first-prompt title extension.** It initially titles from the prompt, then reevaluates every four turns; bulk operations can send full session contents. It prefers lightweight models (`gemini-flash-lite-latest`, Claude Haiku, then GPT mini, then the active model), has a 15-second timeout, and pauses auto-retitling after manual `/name`. Version `0.8.0` is documented in its Pi gallery assets and the package includes test and smoke-test workflows, but repeated calls and broad indexing/search scope put it outside this repository’s intended minimal behavior. [Pi gallery](https://pi.dev/packages/pi-sessions) · [npm metadata/README](https://www.npmjs.com/package/pi-sessions)

11. **Official Pi currently provides naming primitives and a manual example, not automatic title generation.** The official example registers `/session-name` to call `pi.setSessionName()` or show `pi.getSessionName()`. Official session-format docs say names can be set by `/name`, `--name`/`-n`, or extensions, and are displayed in `/resume`. A still-open feature request proposes built-in fire-and-forget automatic naming after the first successful assistant response, explicitly preserving existing names, but it is a proposal rather than shipped evidence. [Official example](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/examples/extensions/session-name.ts) · [official session format](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/session-format.md#sessioninfoentry) · [official feature request #2090](https://github.com/earendil-works/pi/issues/2090)

12. **Terminal titles, summaries, and manual rename tools are adjacent, not exact competitors.** `pi-title-renamer` changes only the terminal tab after the first assistant reply; it is not persisted session naming. Pi’s official `/name`/`setSessionName` facilities are manual primitives. Compaction/branch summaries are different session content types. Packages whose main behavior is `/rename` or `/auto-session-name` on demand are manual tools unless they also register an automatic lifecycle handler. [Terminal-title package](https://github.com/mkioutcc/pi-title-renamer) · [official session entry types](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/session-format.md) · [official manual example](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/examples/extensions/session-name.ts)

## Gaps

- The Pi gallery is paginated and its search/indexing was inconsistent; absence from the gallery cannot prove that another new package does not exist. npm and GitHub search likewise cannot prove completeness, especially for unindexed/private packages or renamed forks.
- Download counts and publish dates were only available for some gallery entries; GitHub source structure and package versions are used as maturity proxies, not adoption proof.
- No live latency, token-usage, or billing benchmark was run. Latency/cost statements above are either direct code properties (model, caps, timeout, detached promise) or clearly identified package-author claims.
- Model-role configuration is user-dependent. A package saying “utility,” “active,” “auto,” or “Haiku” does not guarantee a particular price or latency for every installation.
- Source was inspected at each repository’s current default branch, not pinned release tarballs, except where the gallery exposed a concrete version. A published npm artifact can lag its repository.
